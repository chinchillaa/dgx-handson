# ML ハンズオンセッション 設計方針ドキュメント

> **このドキュメントの用途**
> Claude Code が各章のコンテンツ（スライド・演習コード・環境構築スクリプト・課題シートなど）を実装するための設計仕様書。
> 実装の判断に迷った場合はこのドキュメントに立ち返ること。

---

## 1. カリキュラム全体のコンセプト

### 1.1 基本方針

「理論を教えてから手を動かす」ではなく、**「動かしながら理論を理解する」** 順序を徹底する。
各トピックは必ず「動くコードが先、説明は後」の構成にする。

### 1.2 到達目標

| フェーズ | 到達目標 |
|---|---|
| 第1章終了時 | LLMが「何をしているか」を自分の言葉で説明できる。推論パラメータを意図的に調整できる。 |
| 第2章終了時 | タスクに応じてSFT・DPO・RAGを使い分けられる。学習を実際に回してログを読める。 |
| 第3章終了時 | Tool use付きエージェントを設計・実装し、マルチエージェント構成に拡張できる。 |

### 1.3 一貫したナラティブ

3章を通じて「1つのプロダクト」を育てる構成にする。

- 第1章：素のLLMで問題を解かせ、限界を確認する
- 第2章：SFT・RAGでそのLLMを強化する
- 第3章：強化したLLMをエージェントの頭脳として組み込む

**具体的なプロダクト例（実装時に選択すること）**

以下のいずれかを推奨。章をまたいで同じデータ・タスクを使いまわせるものを選ぶ。

- 社内ドキュメントQAエージェント（RAGとの親和性が高い）
- コードレビュー支援エージェント（エンジニア参加者に響く）
- 日本語ニュース要約・検索エージェント（日本語データが入手しやすい）

## 2. 参加者の前提スキルと期待値設定

### 2.1 前提スキル（入室条件）

以下をすべて満たすことを前提とする。コンテンツはこのレベルに合わせて設計する。

- Python の基本文法（関数・クラス・リスト内包表記）
- NumPy / pandas の基本操作
- 線形代数の基礎（行列積・ベクトルの概念程度）
- Git の基本操作（clone・commit・push）
- Linux コマンドラインの基本（ls・cd・ssh・tmux 程度）

### 2.2 前提としないスキル

以下は前提にしない。必要な概念はその場で説明する。

- PyTorchの使用経験
- 深層学習の数学的な詳細（偏微分・バックプロパゲーションの導出など）
- HuggingFace Transformers の使用経験
- Docker・Kubernetes の運用知識

### 2.3 期待値設定の注意点

- 「LLMが作れるようになる」ではなく「LLMを正しく使いこなせるようになる」が目標である
- 第2章のSFTは「学習を完走させること」が目的であり、モデル精度の最大化は目的外
- 数式の導出は行わない。「なぜこの式になるか」より「この式が何を意味するか」を優先する

## 3. 各章の設計方針と選定理由

### 第1章：機械学習(AI)の仕組み

#### 設計方針

- 抽象度を段階的に上げる：線形回帰 → NN → Transformer → LLM の順で積み上げる
- 各ステップで「前のステップの何が不足していたか」を明示し、拡張の必然性を示す
- 「LLMは関数である」という視点を第1章全体の軸にする

#### トピックと選定理由

| トピック | 選定理由 |
|---|---|
| 損失関数・勾配降下法 | 「学習」の本質であり、第2章のSFT理解に直結するため |
| ニューラルネットワーク | Transformerの前提知識。層・重み・活性化関数の概念を固める |
| Transformer / Attention | 現代LLMの根幹。Self-attentionの直感（どのトークンに注目するか）を押さえる |
| 事前学習の概念 | SFTとの対比で理解できるよう第1章で導入しておく |
| 推論パラメータ（temperature等） | 最初のハンズオンで即実感できる。参加者の興味を引くエントリーポイント |

#### ハンズオン設計

1. **NumPyで線形回帰（勾配降下）の手実装**
   - `sklearn` 等のライブラリを使わず、forループで重みを更新する
   - 損失曲線を `matplotlib` でリアルタイムプロットする
   - 目的：「学習とは何か」を手で動かして体感させる

2. **PyTorchで2層NNによるMNIST分類**
   - `nn.Module` の最小構成に絞る
   - Epochごとに loss と accuracy を記録・可視化する
   - 目的：自動微分とミニバッチ更新の流れを体感させる

3. **HuggingFace Transformersで既存LLMの推論**
   - モデルは `elyza/ELYZA-japanese-llama-2-7b` 等の日本語LLMを使用
   - `temperature` / `top_p` / `max_new_tokens` を変えながら出力差を観察する
   - 目的：パラメータと出力の関係を経験的に把握させる

### 第2章：AIを「強く」する

#### 設計方針

- 「どの手法をいつ使うか」の判断フレームワークを章の冒頭に提示する
- 手法間の比較（SFT vs RAG 等）は、同一の入力・タスクで出力を並べて評価させる
- 評価を「感覚」ではなく「指標」で行う習慣をつける

#### 手法選定の判断フレーム（実装すること）

```
Q1: タスク固有の知識が必要か？
  → YES → Q2: 知識は動的（頻繁に更新）か？
    → YES → RAG
    → NO  → SFT が有力
  → NO  → Prompt Engineering または DPO（応答スタイル調整）
```

#### トピックと選定理由

| トピック | 選定理由 |
|---|---|
| SFT (Supervised Fine-Tuning) | タスク特化の最も直接的な手法。「データが全て」という感覚を体感させる |
| LoRA | DGX上でも現実的なVRAM内に収める必須テクニック |
| DPO | RLHFの簡略版。preference datasetの作り方まで体験させる |
| RAG | 知識の外部化という概念。SFTとの使い分けを明確にする |
| 評価指標 | BERTScore・ROUGE・LLM-as-judge の三種を実際に計算させる |

> **RLHFを直接扱わない理由**：PPOは報酬モデルの学習が別途必要でDGX 1台では時間内に完走が難しい。DPOで「フィードバックを学習に反映する」概念はカバーできる。

#### ハンズオン設計

1. **LLaMA 3 + LoRA + SFTTrainerによるSFT**
   - データセット：HuggingFace Hub から `kunishou/databricks-dolly-15k-ja` 等を使用
   - LoRAのrank・alphaを変えて学習コストと精度のトレードオフを観察する
   - 学習は `nohup` またはtmuxセッションで実行し、`wandb` でログを監視する

2. **DPOのpreference dataset作成と学習**
   - chosen / rejected ペアを手動で5〜10件作成させる（この作業自体が学習）
   - `trl.DPOTrainer` で学習し、SFTモデルとの出力差を並べる

3. **LangChain + ChromaDBによるRAGパイプライン**
   - ドキュメント：参加者が持ち込んだPDFまたは用意した技術文書を使用
   - チャンクサイズ（256 / 512 / 1024 token）を変えて検索精度を比較する
   - Retrieverのtop-k件数と回答品質の関係を実験させる

4. **評価実験**
   - 同一質問セット（10問）に対して「素のLLM」「SFT後」「RAGあり」で回答させる
   - BERTScoreとLLM-as-judgeで採点し、手法間の差を定量化する

### 第3章：AIエージェントを作ろう

#### 設計方針

- エージェントを「LLM + ループ + ツール」として構造的に定義することから始める
- 実装は LangGraph を使用し、グラフ構造でエージェントの動作フローを可視化する
- 「壊れる体験」を意図的に入れる（エラーリカバリ課題）

#### アーキテクチャパターンと選定理由

| パターン | 説明 | 扱う理由 |
|---|---|---|
| ReAct | Reasoning + Acting のループ | エージェントの基本形。最初に実装させる |
| Plan-and-Execute | 計画フェーズと実行フェーズの分離 | マルチエージェントの前段として理解させる |
| Multi-agent | Planner + Executor の役割分担 | 第3章の最終到達点 |

#### トピックと選定理由

| トピック | 選定理由 |
|---|---|
| Tool use / Function calling | エージェントの能力範囲を決める最重要概念 |
| メモリ（Short/Long-term） | 会話の状態管理。実装でよく詰まるポイント |
| エラーリカバリ | 本番で最も重要だが教材で扱われにくい |
| 観測性（ログ・トレース） | LangSmithまたはOpenTelemetryで可視化させる |
| コスト管理 | トークン数・API呼び出し回数のカウント手法 |

#### ハンズオン設計

1. **LangGraphでReActエージェントの構築**
   - ツール：DuckDuckGo検索 + Python REPL + ファイル読み書き
   - グラフの `StateGraph` を図示しながら実装する
   - 目的：「エージェントがどこで何を決定しているか」を構造として見せる

2. **外部API（気象API・模擬DBなど）のtool登録**
   - APIキーは環境変数で管理するパターンを徹底させる
   - エージェントが自律的に正しいツールを選ぶことを確認する

3. **2エージェント構成（Planner + Executor）**
   - タスク：「指定したテーマでレポートを自動生成する」
   - Plannerがタスクを分解 → Executorが各ステップを実行 → 結果をまとめる
   - 第2章で強化したLLMをExecutorとして組み込む（章間の接続）

4. **エラーリカバリ課題（意図的バグ入り）**
   - バグの種類：誤った引数名・存在しないエンドポイント・型不一致
   - エージェントが自己修正できる条件と失敗する条件を観察・考察させる

## 4. DGX環境・インフラ設計方針

### 4.1 前提環境

| 項目 | 仕様 |
|---|---|
| サーバー | NVIDIA DGX（A100 x8 想定） |
| OS | Ubuntu 22.04 |
| アクセス方法 | 参加者各自がSSHで接続 |
| ストレージ | `/home/<username>/` に各自の作業ディレクトリ |
| 共有ストレージ | `/data/shared/` に事前ダウンロード済みモデル・データセットを配置 |

### 4.2 ユーザー分離方針

- 参加者ごとに専用のLinuxユーザーを発行する（例：`user01` 〜 `user20`）
- Dockerコンテナを使用せず、**uv仮想環境（`.venv`）で分離**する
  - 理由：コンテナ内からGPUを扱う設定コストを省く。condaより高速で依存解決が確実
- パッケージ管理は **`uv`** に統一する（`pip` の直接使用は禁止）

```bash
# 各ユーザーの環境セットアップ例（事前に講師側が準備するスクリプト）
# uvのインストール（未インストールの場合）
curl -LsSf https://astral.sh/uv/install.sh | sh
source $HOME/.local/bin/env  # PATHを反映

# 仮想環境の作成と有効化
uv venv .venv --python 3.11
source .venv/bin/activate

# パッケージのインストール
uv pip install torch transformers datasets trl peft langchain langchain-community \
               chromadb sentence-transformers wandb langgraph
```

### 4.3 GPU割り当て方針

| 章 | GPU使用 | 割り当て方針 |
|---|---|---|
| 第1章 | 軽微（MNIST程度） | 全参加者が同時使用可 |
| 第2章（SFT） | 高負荷（A100 40GB x1 以上） | **順番制またはジョブキュー**（後述） |
| 第2章（RAG） | CPUのみ可 | 制限なし |
| 第3章 | 推論のみ（中程度） | 2〜4人で1GPU共有 |

### 4.4 ジョブスケジューリング

第2章のSFTは参加者が同時実行するとVRAMが枯渇する。以下のいずれかで対処する。

**オプションA：手動順番制（推奨・シンプル）**

```bash
# 共有のキューファイルで管理
echo "$(date) $USER" >> /data/shared/gpu_queue.txt
# 自分の順番になったらGPU_IDを指定して実行
CUDA_VISIBLE_DEVICES=0 python train_sft.py
```

**オプションB：Slurmによるジョブキュー（環境があれば）**

```bash
sbatch --gres=gpu:1 --time=01:00:00 train_sft.py
squeue  # 状況確認
```

### 4.5 モデル・データセットの事前配置

ネットワーク帯域を節約するため、以下を事前にダウンロードして共有ストレージに配置する。

```
/data/shared/
├── models/
│   ├── meta-llama/Meta-Llama-3-8B/          # 第2章SFT用
│   ├── elyza/ELYZA-japanese-llama-2-7b/     # 第1・2章推論デモ用
│   └── sentence-transformers/all-MiniLM-L6-v2/  # 第2章RAG用
└── datasets/
    ├── kunishou/databricks-dolly-15k-ja/    # 第2章SFT用
    └── handson-docs/                        # 第2章RAG用ドキュメント群
```

HuggingFace のキャッシュを共有パスに向ける設定を `.bashrc` に追加する。

```bash
export HF_HOME=/data/shared/hf_cache
export TRANSFORMERS_CACHE=/data/shared/hf_cache
```

### 4.6 tmux の使用を標準化

長時間かかるSFT学習はSSH切断に備えて `tmux` セッション内で実行させる。
セッション名の命名規則を統一することで講師がモニタリングしやすくする。

```bash
# 参加者は以下の命名規則でセッションを作成する
tmux new -s sft-train   # SFT学習用
tmux new -s rag-dev     # RAG開発用
```

### 5.2 Notebookの実装規約

- セルは「説明 → コード → 出力確認」の3セットを1単位とする
- コードセルは単独で実行可能にする（前のセルの副作用に依存させない）
- 課題セルには `# TODO:` コメントで穴埋め箇所を明示する
- 実行時間の目安をセルの冒頭コメントに記載する（例：`# 実行時間: 約3分`）

### 5.3 講義コンテンツの実装規約

- 講義コンテンツはすべて **HTMLファイル** として実装する（Marpは使用しない）
- 単一HTMLファイルで完結させる（外部サーバー不要・ブラウザで直接開ける）
- スタイルは外部CDN（Tailwind CSS等）を使用してよい
- コードスニペットは必ず実行済みの出力と並べて掲載する
- コードブロックにはシンタックスハイライト（highlight.js等）を適用する
- アニメーション・動的エフェクトは使用しない（印刷・PDF出力を想定する）
- 数式が必要な箇所は MathJax または KaTeX を CDN 経由で使用する

### 5.4 実装時の禁止事項

- `openai` パッケージに依存したコードを書かない（HuggingFaceとLangChainに統一）
- 外部APIキーをコードにハードコードしない（すべて環境変数経由）
- 参加者のGPU環境を前提としたコード（`device="cuda"` の固定記述）を書かない
  - 代わりに `device = "cuda" if torch.cuda.is_available() else "cpu"` を使用する
- モデルを毎回ダウンロードするコードを書かない（`/data/shared/` のキャッシュを参照）
- `pip` コマンドを直接使用したコード・スクリプト・手順書を書かない
  - パッケージのインストールは必ず `uv pip install` を使用する
  - requirements.txtからのインストールは `uv pip install -r requirements.txt` を使用する

### 5.5 ブラウザ表示コンテンツ（web/）の実装規約

各章の `web/` ディレクトリに以下の3種類のHTMLファイルを作成する。

#### ファイル種別と役割

| ファイル | 役割 |
|---|---|
| `index.html` | 章全体の進行ガイド。目次・所要時間・各トピックへのリンクを含む |
| `supplement_<topic>.html` | 技術補足。Notebookに収まらない概念説明・背景知識・図解を提供する |
| `quiz_<topic>.html` | 理解確認問題集。各トピック終了後に参加者が自己採点できる形式にする |

#### index.html の必須要素

- 章の概要と到達目標
- セクション一覧と推奨所要時間
- 各 `supplement_*.html` / `quiz_*.html` へのリンク
- 関連 Notebook ファイル名の明示
- 「次の章へ」リンク（章間のナビゲーション）

#### supplement_*.html の実装方針

- 1ファイル1トピックに限定する（スクロール量を抑える）
- 説明 → コード例 → 図解 の順で構成する
- コードブロックは highlight.js でシンタックスハイライトを適用する
- 数式が必要な場合は KaTeX（CDN）を使用する

#### quiz_*.html の実装方針

- 選択式・穴埋め式・記述式を混在させる
- 解答はHTMLソース内に埋め込み、ボタン押下で表示する（JavaScriptで制御）
- 解説文を必ず付ける（正解・不正解の理由を説明する）
- 採点ロジックはすべてクライアントサイドJavaScriptで完結させる

#### 共通スタイル・技術スタック

- CSSフレームワーク：Tailwind CSS（CDN）
- シンタックスハイライト：highlight.js（CDN）
- 数式レンダリング：KaTeX（CDN、必要な箇所のみ）
- フォント：システムフォントスタックを使用（外部フォントCDN不要）
- 外部CDN以外の依存は持たない（単体で動作すること）
