前置き

推論パラメータとは何か

前の補足資料で「LLM は次トークンの確率分布を出力する」と学びました。 確率分布が得られた後、どのトークンを実際に選ぶかを制御するのが推論パラメータです。 モデルの重みは変えずに、出力の傾向——創造性・多様性・一貫性——を外側から調整できます。

トークン生成の流れ

Transformer 順伝播
コンテキストを処理し、語彙サイズ分の logit ベクトルを出力
↓
Temperature 適用
logit を Temperature で割ることで分布の形状を変える
↓
Top-k / Top-p フィルタ
候補を絞り込み、低確率トークンを除外する
↓
サンプリング
残った確率分布から 1 トークンを確率的に選択
↓ 繰り返す

Greedy 選択との違い

最も確率が高いトークンを常に選ぶ「Greedy デコーディング」は決定的（同じ入力から同じ出力）ですが、 繰り返しや退屈な表現が生まれやすくなります。 サンプリングを使うと毎回異なる出力が生成され、創造的なタスクに向きます。

Section 1

Temperature——分布の「シャープさ」を制御する

Temperature $T$ は softmax に渡す前に logit を割ることで、 確率分布の形状を変えます。 低いと分布が鋭く（確実）、高いと分布が平たく（多様）なります。

Temperature つき softmax

$$P(x_i) = \frac{\exp(z_i / T)}{\displaystyle\sum_j \exp(z_j / T)}$$
$z_i$ トークン $i$ の logit（Transformer の生スコア）
$T \to 0$ 最大 logit のトークンに確率 1 が集中 → Greedy 選択と等価
$T = 1$ logit をそのまま softmax にかける標準動作
$T \to \infty$ 全トークンの確率が均一になり、ランダム選択と等価

同じ logit に異なる Temperature を適用した結果

Temperature の推奨設定

値の目安 特性 適したユースケース
0.0 – 0.3 確実・再現性高 コード生成・ファクト抽出・分類
0.7 – 1.0 バランス型 一般的な会話・要約・翻訳
1.2 – 2.0 多様・創造的 創作・ブレインストーミング・詩

Section 2

Top-p と Top-k——候補を絞り込む

Temperature が分布の形を変えるのに対し、Top-p と Top-k は サンプリングの候補集合そのものを絞り込むフィルタです。 低確率の「外れ値トークン」が選ばれるのを防ぎ、出力品質を安定させます。

Top-k サンプリング

確率の高い上位 $k$ 個のトークンだけを残し、それ以外を除外してから再正規化してサンプリングします。

特徴: 候補数が常に固定。分布が広い場合でも低確率トークンが混入しにくい。

Top-p サンプリング（Nucleus）

確率の高いトークンから順に累積して、累積確率が $p$ を超えた時点以降を除外します。

特徴: 候補数が分布に応じて動的に変化。確信度が高い時は少数候補から選ぶ。

Top-p と Top-k の使い分け

多くの実装では Top-p（Nucleus Sampling）が推奨されます。 確率分布の形状に応じて候補数が自動調整されるため、 「鋭い分布のときは決定的に、広い分布のときは創造的に」という動作が自然に得られます。 Top-k は高速な推論が求められる場面で使われます。

Section 3

その他の主要パラメータ

max_new_tokens 生成する最大トークン数

生成を打ち切るトークン数の上限です。小さすぎると文章が途中で切れ、 大きすぎると不要な繰り返しや「埋め草」が生まれやすくなります。 タスクに合わせて設定します（要約: 128〜256、コード: 512〜2048）。

repetition_penalty 繰り返しへのペナルティ

直近に出現したトークンの logit を割り引くことで、同じ表現の繰り返しを抑制します。

$$z'_i = \begin{cases} z_i / r & \text{if } x_i \text{ は既出トークン} \\ z_i & \text{otherwise} \end{cases}$$

$r = 1.0$（変化なし）〜 $r = 1.3$（強めのペナルティ）。大きくしすぎると文が不自然になります。

do_sample サンプリングの有効化

False（デフォルト）のとき Greedy デコーディングになります。 Temperature / Top-p / Top-k を有効にするには do_sample=True が必要です。

HuggingFace generate() の典型的な設定例 
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

model_name = "meta-llama/Llama-3.2-1B-Instruct"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name, torch_dtype=torch.float16
).to("cuda")

prompt = "深層学習とは何か、一言で説明してください。"
inputs = tokenizer(prompt, return_tensors="pt").to("cuda")

# ── 創造的なタスク向け設定 ──────────────────────────────
outputs_creative = model.generate(
    **inputs,
    max_new_tokens=200,
    do_sample=True,
    temperature=1.1,       # 高温 → 多様な表現
    top_p=0.92,            # 上位 92% の累積確率から選択
    repetition_penalty=1.1,
)

# ── 正確さが求められるタスク向け設定 ─────────────────────
outputs_factual = model.generate(
    **inputs,
    max_new_tokens=200,
    do_sample=True,
    temperature=0.2,       # 低温 → 確実・再現性高
    top_p=0.9,
    repetition_penalty=1.0,
)

print(tokenizer.decode(outputs_creative[0], skip_special_tokens=True))
print(tokenizer.decode(outputs_factual[0],  skip_special_tokens=True))

Section 4

パラメータの組み合わせ方針

パラメータは互いに影響し合います。 「Temperature だけ下げれば品質が上がる」という単純な話ではなく、 タスクの性質に合わせてセットで調整するのが基本です。

タスク別推奨パラメータセット

タスク Temperature Top-p rep. penalty
コード生成・バグ修正 0.1 – 0.3 0.90 1.0
ファクト抽出・要約 0.2 – 0.5 0.90 1.1
一般的な Q&A・会話 0.7 – 0.9 0.92 1.1
創作・ブレスト 1.0 – 1.3 0.95 1.15
多様な候補を列挙 1.2 – 1.5 0.98 1.2

※ 最適値はモデルによって異なります。ハンズオン③で実際に試してみましょう。

よくある失敗パターン

  • Temperature を上げすぎる（≥ 2.0）→ 意味をなさないトークン列が生成される
  • Top-p と Top-k を両方設定する → 制約が二重になり候補が極端に絞られる場合がある
  • do_sample=False のまま Temperature を変える → Greedy のため Temperature は無効になる

まとめ

  • 01Temperature は logit を割ることで分布形状を変える——低いと確実、高いと多様
  • 02Top-p（Nucleus）は累積確率で候補を動的に絞る——分布の形に適応するため汎用性が高い
  • 03パラメータはタスクの性質に合わせてセットで調整する——単独で最大化・最小化するものではない
  • 04ハンズオン③では実際にパラメータを変えて出力の変化を体感する
